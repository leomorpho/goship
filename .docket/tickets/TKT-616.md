# TKT-616: Make ship-mcp execute the checked-out ship CLI instead of relying on a stale PATH binary.

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-616` before committing. -->

## Description
Objective: Make ship-mcp execute the checked-out ship CLI instead of relying on a stale PATH binary.

Context:
The current MCP server shells out to `ship` on PATH. That breaks the core LLM promise because the globally installed binary can drift from the checked-out repo and advertise removed or stale command surfaces.

Rails/Laravel DX bar:
- LLM tooling must reflect the current checkout, not ambient machine state.
- If the repo provides an MCP server, it should be more reliable than copy-pasting shell commands.

Likely paths:
- tools/mcp/ship/internal/server/tools_ship.go
- tools/mcp/ship/internal/server/tools.go
- tools/mcp/ship/internal/server/server_test.go
- tools/mcp/ship/internal/server/tools_test.go
- tools/mcp/ship/README.md

Likely symbols:
- lookPathShip
- runShipJSON
- runShipDescribePayload
- repo-root CLI resolution

Tests to add or update:
- MCP server tests proving repo-local CLI resolution wins over PATH drift.
- Failure-mode coverage for missing repo-local CLI/module-path execution.

Verify commands:
- docket validate TKT-612
- go test ./tools/mcp/ship/... -count=1
- go run ./tools/mcp/ship/cmd/ship-mcp

Out of scope:
- Adding new MCP tools.
- Rewriting MCP docs beyond what is required to explain the new resolution model.
- Changing the GoShip CLI surface itself.
