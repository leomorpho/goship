# TKT-671: extend ship route and endpoint metadata so frontend contract codegen has real inputs

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-671` before committing. -->

## Description
Objective: extend ship route and endpoint metadata so frontend contract codegen has real inputs.

Context:
Current route inventory is useful for discovery but too shallow for trustworthy frontend generation. This ticket should define and emit the additional machine-readable endpoint metadata needed to support generated frontend contracts without reviving framework-owned OpenAPI as the main product surface.

Rails/Laravel DX bar:
- Prefer one deterministic contract export over ad hoc reflection or fragile scraping.
- Generated frontend artifacts should come from explicit backend metadata, not from LLM guesswork around handlers.

Likely paths:
- tools/cli/ship/internal/commands/routes.go
- tools/cli/ship/internal/commands/describe.go
- framework/api/
- docs/reference/01-cli.md
- docs/guides/08-building-an-api.md

Likely symbols:
- route inventory JSON
- endpoint operation ids
- request and response contract metadata
- auth requirement metadata
- contract export command or extended JSON shape

Tests to add or update:
- CLI contract tests for emitted frontend-contract metadata.
- Any framework/api contract tests needed to keep exported metadata aligned with the JSON envelope and error semantics.

Verify commands:
- docket validate TKT-671
- go test ./tools/cli/ship/internal/commands ./framework/api -count=1

Out of scope:
- Full TypeScript package generation and reference-app integration.
- Reintroducing OpenAPI generation or serving as the default contract path.
- Supporting arbitrary downstream frontend tools.

## Acceptance Criteria
- [ ] GoShip emits machine-readable endpoint metadata that goes beyond method/path/auth/handler and is sufficient input for generated frontend contracts.
- [ ] Automated tests freeze the emitted metadata shape and keep it aligned with framework API envelope and error conventions.
- [ ] Verification commands and outcomes are recorded in ticket notes or handoff.
