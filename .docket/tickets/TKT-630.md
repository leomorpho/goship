# TKT-630: Inject the canonical project-context bundle into ship agent:start and ship-mcp so agents receive task-ready context automatically.

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-630` before committing. -->

## Description
Objective: Inject the canonical project-context bundle into `ship agent:start` and `ship-mcp` so agents receive task-ready context automatically.

Context:
A context bundle does not solve the LLM usability problem if models still have to know it exists. The main agent entrypoints should hand the same context bundle to agents automatically, so the supported path is push-based instead of discovery-based.

Rails/Laravel DX bar:
The first-party agent workflow should arrive with the same trustworthy context every time, without a weaker model needing to guess which repo command or MCP tool to call first.

Likely paths:
- tools/cli/ship/internal/commands/agent_start.go
- tools/cli/ship/internal/commands/*agent*_test.go
- tools/mcp/ship/internal/server/tools.go
- tools/mcp/ship/internal/server/tools_ship.go
- tools/mcp/ship/internal/server/tools_test.go
- docs/reference/01-cli.md
- docs/reference/02-mcp.md

Likely symbols:
- `ship_context`
- `TASK.md`
- `ship agent:start`
- `ship-mcp`
- `context-bundle-v1`

Tests to add or update:
- `ship agent:start` tests proving `TASK.md` includes the canonical context bundle
- MCP tests proving the same bundle is exposed through the supported agent path
- docs sync tests if the entry workflow contract changes

Verify commands:
- docket validate TKT-630

Out of scope:
- Building an agent orchestration system inside GoShip.
- Adding support for arbitrary third-party agent conventions beyond the first-party entry surfaces.
- Replacing the existing CLI/MCP commands behind the bundle.
