# TKT-654: eliminate stale persisted-age behavior in profiles

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-654` before committing. -->

## Description
modules/profile/profile.go stores a calculated age on create/update and carries a TODO admitting the value becomes stale unless a separate task updates birthdays daily. Even if this is app-domain logic, it is a correctness bug: age should not silently drift as time passes.

Fix:
1. Decide whether age should be derived at read time or refreshed by a scheduled task
2. Implement the chosen approach so stored/displayed age stays correct across birthdays
3. Add tests around birthday boundaries

Likely paths:
- modules/profile/profile.go
- modules/profile/*_test.go
- any job/scheduler wiring if a refresh task is chosen

Out of scope: redesigning dating or matching product logic beyond age correctness.

Verify commands:
- go test ./modules/profile/...
- Manual: verify a profile crossing a birthday boundary reflects the correct age

## Acceptance Criteria
- [ ] profile age no longer goes stale after birthdays
- [ ] tests cover birthday boundary behavior
- [ ] the chosen derive-vs-refresh approach is explicit in code
- [ ] go test ./modules/profile/... passes
