# TKT-659: classify every first-party module candidate as core, battery, starter-app, or delete

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-659` before committing. -->

## Description
Before rewriting code, GoShip needs one explicit disposition for every module candidate. The current confusion comes from treating all packages under modules/ as if they deserve the same future. This ticket creates the authoritative keep/rewrite/eject map.

Deliver:
1. A module inventory covering current packages in modules/ plus adjacent battery-like surfaces
2. A disposition for each: core seam, installable battery, starter/example app code, or delete
3. A short rationale and intended destination for each package
4. The list wired into docs and/or machine-readable metadata used by ship tooling where appropriate

Likely paths:
- docs/architecture/ or docs/reference/ module docs
- config/modules.yaml and battery catalog docs if they need to reflect canonical first-party status
- tools/cli/ship/internal/commands/describe.go and runtime report docs if the first-party battery baseline changes

Out of scope: implementing the rebuilds themselves.

Verify commands:
- ship describe --pretty
- ship verify --profile strict
