# TKT-595: [Flags Layer B] Implement idempotent startup syncer

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-595` before committing. -->

## Description
Objective: On app startup, sync the code-defined flag registry into the DB — creating missing rows, refreshing descriptions, and never touching the `enabled` state that admins control.

Context:
After TKT-594, flags are defined in code as typed constants. But they don't exist in the DB until someone creates them manually. The syncer eliminates that manual step. It runs once during the flags module's `Start` lifecycle hook (or equivalent bootstrap point) and is fully idempotent — safe to run on every deploy.

Sync logic:
1. Call `flags.All()` to get all registered definitions.
2. Fetch all existing flag rows from the DB (one query).
3. For each registered definition:
   - If no DB row exists → `INSERT` with `key`, `description`, `enabled = definition.Default`.
   - If DB row exists → `UPDATE description` only. Never touch `enabled`, `rollout_pct`, or `user_ids`.
4. Log a summary at INFO level: N created, M updated, K unchanged.

The syncer must not delete orphaned DB rows — that's a display concern handled in TKT-599.

## Acceptance Criteria
- [ ] Syncer creates flag rows for newly registered definitions on startup.
- [ ] Syncer updates description when it changes in code, without touching `enabled`.
- [ ] Syncer is idempotent — running it N times has the same effect as running it once.
- [ ] Sync result (created/updated/unchanged counts) is logged at INFO level.
- [ ] All syncer unit tests pass (`go test ./modules/flags/... -run TestSync -count=1`).
- [ ] Code changes are limited to the described scope.
