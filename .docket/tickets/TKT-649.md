# TKT-649: standardize storage repo image APIs and bucket ergonomics

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-649` before committing. -->

## Description
framework/repos/storage/storagerepo.go carries two API-design TODOs that matter for reuse as a framework package.

Confirmed issues:
- Bucket is an unadorned int enum with a TODO to adopt stronger enum ergonomics
- GetImageObjectFromFile() and GetImageObjectsFromFiles() duplicate logic and expose an awkward single-vs-many split, with presigned URL expiry hard-coded in both flows

Fix:
1. Define a clearer bucket API (stringer/parse helpers or another explicit typed contract)
2. Consolidate single and bulk image hydration behind shared logic
3. Move presigned URL expiry policy to a centralized setting or parameter rather than burying it in one code path
4. Update tests and call sites to the refined API

Likely paths:
- framework/repos/storage/storagerepo.go
- framework/repos/storage/*_test.go
- modules/profile/ and other image consumers

Out of scope: changing storage backends or object-key layout.

Verify commands:
- go test ./framework/repos/storage/...
- go test ./...

## Acceptance Criteria
- [ ] storage bucket selection uses a clearer typed contract than the current loose int enum
- [ ] single-file and multi-file image hydration share one internal implementation
- [ ] presigned URL expiry policy is centralized or configurable instead of hard-coded in one helper
- [ ] storage tests updated for the refined API
- [ ] go test ./framework/repos/storage/... passes
- [ ] go test ./... passes
