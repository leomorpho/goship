# TKT-655: move notification enums out of framework/domain and fix platform naming

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-655` before committing. -->

## Description
framework/domain/enum.go still owns notification-specific enums with TODOs to move them into the notifications module, and one of the platform values is explicitly named in a confusing way. That leaks product-specific notification concerns into the shared domain package and makes the public API harder to reason about.

Confirmed issues:
- NotificationPermissionType and NotificationPlatform carry TODOs to move into notifications
- NotificationPlatformPush actually means PWA push, while FCM already has its own distinct value
- notifier.go still carries adjacent TODOs around notification typing semantics, which reinforces that the current ownership boundary is muddy

Fix:
1. Move notification-specific enums/types to modules/notifications or another notification-owned seam
2. Rename the ambiguous platform constant/value so PWA and FCM are clearly distinguished
3. Update imports/call sites across notifications and any consumers
4. Keep serialization/storage behavior stable or migrate it intentionally

Likely paths:
- framework/domain/enum.go
- modules/notifications/...
- callers that reference notification enums from framework/domain

Out of scope: redesigning notification UX or delivery channels.

Verify commands:
- go build ./...
- go test ./modules/notifications/...
- go test ./...
