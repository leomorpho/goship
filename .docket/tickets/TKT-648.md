# TKT-648: improve profile service test strategy and assertions

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-648` before committing. -->

## Description
modules/profile/profile_test.go contains multiple TODOs that indicate weak confidence rather than missing features. The current suite restarts Postgres-backed infrastructure for many scenarios, leaves key assertions shallow, and comments out mock expectation verification when behavior is hard to prove.

Confirmed gaps:
- move most scenarios to SQLite-backed tests and keep a smaller Postgres/pgvector contract run
- expand GetProfileByID assertions beyond ID equality
- verify or explicitly document friend ordering semantics
- replace debugger-only confidence in storage mock interactions with reliable assertions or a better fake

Fix:
1. Split fast unit coverage from heavier integration coverage
2. Strengthen object field assertions in the profile read tests
3. Add a stable test for friend ordering or codify that ordering is unspecified
4. Make image upload/profile photo tests assert storage interactions in a repeatable way

Likely paths:
- modules/profile/profile_test.go
- modules/profile/profile.go
- framework/tests/ (if new helpers are needed)
- framework/repos/storage/ test doubles used by the profile tests

Out of scope: redesigning profile product behavior.

Verify commands:
- go test ./modules/profile/...
- go test -tags integration ./modules/profile/...

## Acceptance Criteria
- [ ] core profile service scenarios run in fast unit tests where possible
- [ ] GetProfileByID tests verify meaningful field-level behavior
- [ ] friend ordering behavior is either tested explicitly or documented as intentionally unspecified
- [ ] profile image/storage tests use reliable assertions instead of debugger-only comments
- [ ] go test ./modules/profile/... passes
- [ ] go test -tags integration ./modules/profile/... passes
