# TKT-660: define and enforce the canonical standalone battery contract

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-660` before committing. -->

## Description
Installable batteries need one public shape. Today some modules expose Module types, some only expose services, some split route registration into sibling packages, and some depend directly on app-domain packages. This ticket defines the canonical contract and wires it into generator/policy/tooling so new batteries cannot regress.

Contract goals:
- one public entrypoint
- one canonical runtime/config deps shape
- routes/migrations/jobs owned by the same battery package when applicable
- no app-domain imports across the battery boundary
- consistent standalone module packaging and go.work/go.mod expectations

Likely paths:
- tools/cli/ship/internal/generators/module.go
- tools/cli/ship/internal/commands/module.go
- tools/cli/ship/internal/commands/verify.go
- tools/cli/ship/internal/policies/doctor_*
- docs/reference/01-cli.md and architecture docs describing batteries

Out of scope: migrating every existing battery in the same ticket.

Verify commands:
- go test ./tools/cli/ship/internal/...
- ship verify --profile strict
