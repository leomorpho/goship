# TKT-586: Define and document GoShip beta readiness checklist with explicit pass/fail criteria

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-586` before committing. -->

## Description
Define what GoShip beta means with explicit, testable criteria. Inspired by DHH's 15-minute blog demo for Rails 1.0. The checklist must include: (1) ship new creates a production-credible app, (2) all core batteries install and remove cleanly, (3) auth golden flow passes in browser, (4) single-binary path works without Redis/Postgres, (5) ship doctor passes on a fresh clone, (6) getting-started guide completes in under 30 minutes, (7) upgrade path from N-1 to N works without data loss.

Output: docs/beta-readiness.md.

Likely paths:
- docs/beta-readiness.md (new)
- docs/roadmap/01-framework-plan.md

Verify commands:
- docket validate $TICKET_ID

Out of scope:
- Implementing any of the checklist items in this ticket.
- Full matrix E2E coverage for optional modules.

## Acceptance Criteria
- [ ] docs/beta-readiness.md exists with explicit pass/fail criteria for each checklist item.
- [ ] Checklist references the corresponding docket tickets for each criterion.
- [ ] The document is clear enough that any team member can determine beta status without asking the author.
