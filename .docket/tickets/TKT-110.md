# TKT-110: A09 - Managed Mode Config Authority

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-110` before committing. -->

## Description
**Source:** `docs/architecture/09-standalone-and-managed-mode.md`
**Roadmap Group:** Architecture boundary
**Original Status:** `[ ] todo`

**Status:** `[ ] todo`
**Files:** `config/config.go`, `framework/runtimeconfig/` (new), `app/foundation/container.go`, tests under `framework/runtimeconfig/`

**Context:**
GoShip must stay fully standalone while still allowing an external authority to manage a hosted app.
That requires an explicit config resolution model instead of ad hoc environment drift or control-plane
file mutation.

**What to do:**
1. Define a managed-mode config model in GoShip with an allowlisted override surface.
2. Implement config resolution order:
   - framework defaults
   - app repo config
   - environment/secrets
   - managed overrides
3. Make the runtime able to report the effective value and source of managed keys.
4. Add tests covering collision and precedence behavior.

**Done when:** Managed and standalone apps resolve config through the same code path, managed
overrides are explicit and allowlisted, and the runtime can report where each managed value came from.
