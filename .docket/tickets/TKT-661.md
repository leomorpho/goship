# TKT-661: freeze legacy modules tree and define the quarantine boundary

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-661` before committing. -->

## Description
The current mixed modules tree should stop acting as the destination architecture. This ticket establishes the quarantine rule: legacy module code is reference material and bugfix-only unless a ticket explicitly rebuilds or extracts a battery. The purpose is to stop incremental cleanup from preserving accidental architecture.

Deliver:
1. A documented legacy/future boundary for modules code
2. A clear place or naming rule for legacy implementations if files are moved or marked
3. Guardrails in docs and/or policy so new work lands in the new battery surface instead of the legacy tree by default
4. An explicit exception policy for production bug fixes during the transition

Likely paths:
- docs/architecture/ and docs/reference/ module docs
- tools/cli/ship/internal/policies/ or doctor checks if the repo should warn on legacy additions
- modules/ directory layout if a legacy namespace is introduced

Out of scope: performing all module moves in the same ticket.

Verify commands:
- ship verify --profile strict

## Acceptance Criteria
- [ ] the repo has a documented legacy-vs-new battery boundary
- [ ] contributors can tell where new battery work should land
- [ ] legacy module code is explicitly treated as reference or bugfix-only during the transition
- [ ] ship verify --profile strict passes
