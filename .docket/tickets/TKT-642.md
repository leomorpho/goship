# TKT-642: refactor: consolidate RenderPage and RenderToHTMLBlob, fix domain context bug

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-642` before committing. -->

## Description
framework/web/ui/controller.go has two issues:

1. RenderPage() (lines 90-153) and RenderToHTMLBlob() (lines 38-72) duplicate page validation, AppName defaulting, and domain assignment. TODO at line 38 acknowledges this.

2. Domain context bug at line 95: page.Domain is set after the page object is constructed, so components that access domain during rendering see a stale/unset value. Current workaround: manually set page.Domain in every route handler before calling RenderPage.

Fix:
- Extract private preparePage(page Page, domain string) (Page, error) handling validation, AppName defaulting, domain assignment
- Both RenderPage and RenderToHTMLBlob call preparePage first
- Domain bug resolved: preparePage sets domain before any template execution
- Remove per-route domain workarounds once confirmed fixed

Likely paths:
- framework/web/ui/controller.go
- Route handlers that manually set page.Domain as workaround

Out of scope: changes to template rendering engine or Page type definition.

Verify commands:
- go build ./...
- go test ./framework/web/...
- Manual: verify domain visible in both layout navbar and page components (not just layout)
