# TKT-186: Extract framework-level reusable page base types from app UI layer

## Description
Extract framework-level reusable page base types from app UI layer.

Context:
After `app/controller` removal, canonical import is `app/web/ui`, but we still want reusable non-app page contracts at framework level for long-term module/framework ergonomics.

Scope:
- Define a framework-owned page base package (for example `framework/web/page`) for reusable types.
- Move only app-agnostic fields/behavior first.
- Keep app-specific fields/types in app wrapper layer.
- Migrate module/framework consumers to framework base only where boundaries remain clean.

Out of scope:
- Reintroducing compatibility alias packages.
- Forcing app-specific route-name/view enums into framework.

Likely paths:
- `framework/web/**` (new)
- `app/web/ui/page.go`
- `modules/**/views/**` and related Go call sites where appropriate

Verify commands:
- `go test ./framework/... ./app/web/ui/... ./modules/...`
- `go test ./...`

## Acceptance Criteria
- [x] Framework package contains reusable page base types without importing app packages. : Added framework-owned reusable page base package at framework/web/page (base.go, base_test.go); package imports framework/context, htmx, uxflashmessages, Echo/templ only (no app imports).
- [x] app/web/ui wraps/extends framework base for app-specific fields. : Updated app/web/ui/page.go so ui.Page embeds frameworkpage.Base and keeps app-specific fields; NewPage seeds framework base then app-specific data.
- [x] No new framework->app import edges are introduced. : Verified no framework->app edge introduced by this extraction path and compilation stays green; go test ./framework/web/page ./app/web/ui/... ./modules/... and go test ./... passed.
