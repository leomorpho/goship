# TKT-623: Add TypeScript contract artifacts for split frontend apps without reviving framework-owned OpenAPI generation.

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-623` before committing. -->

## Description
Objective: Add TypeScript contract artifacts for split frontend apps without reviving framework-owned OpenAPI generation.

Context:
The real DX gap for a custom frontend is not just JSON responses; it is a trustworthy typed contract that humans and LLMs can consume from TypeScript. GoShip should provide a narrow artifact surface for split frontend apps without falling back to the removed `api:spec` model.

Rails/Laravel DX bar:
- Give custom frontends a contract that is cheaper than hand-maintained fetch code.
- Keep the artifact narrow, deterministic, and aligned with GoShip's actual API helpers and DTO conventions.

Likely paths:
- tools/cli/ship/
- framework/api/
- docs/reference/01-cli.md
- docs/guides/08-building-an-api.md
- starter template paths if the artifact is scaffolded by default for API-only apps

Likely symbols:
- TypeScript contract artifacts
- API DTO/response helper contract
- artifact generation or refresh command surface

Tests to add or update:
- Contract tests for generated TypeScript artifacts.
- CLI/help/docs tests if a new command or generation surface is introduced.

Verify commands:
- docket validate TKT-619
- go test ./tools/cli/ship/internal/commands ./framework/api -count=1

Out of scope:
- Reintroducing framework-owned OpenAPI serving/generation.
- Supporting every possible frontend transport pattern.
- Full SDK generation for external non-TypeScript clients.

## Acceptance Criteria
- [ ] Code changes are limited to the described scope.
- [ ] Automated tests cover the TypeScript contract artifact surface and keep it aligned with the supported API contract.
- [ ] Verification commands and outcomes are recorded in ticket notes or handoff.
