# TKT-607: Document UI provider contract in docs/ui/providers.md

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-607` before committing. -->

## Description
Objective: Write a concise reference doc that explains the UI provider system, available providers, and the explicit limits of the contract.

Context: LLMs and developers need a single authoritative source to understand what GoShip promises and what it does not. This doc is the canonical answer.

Likely files:
- docs/ui/providers.md (new file)

Content to cover:
1. What a UI provider is: a config value set at project init that determines which CSS/JS assets the layout shell loads.
2. Available providers: franken (FrankenUI + UIkit), daisy (DaisyUI + Flowbite), bare (plain Tailwind, no library).
3. How to choose: ship new --ui <provider>. Default is franken.
4. What generated code looks like: bare structural HTML + HTMX wiring + data-* attributes. No library classes.
5. The contract boundary: GoShip only guarantees provider-correct output for freshly generated files. Any file the developer has edited is their responsibility. Provider conversion of hand-edited code is explicitly not supported.
6. Admin module exception: self-contained, not subject to the provider system.

Likely symbols: none (docs only)

Verify commands:
- docket validate TKT-<id>
- cat docs/ui/providers.md

Out of scope:
- Updating any code.
- Documenting internals of how renderBaseLayoutTempl works.

## Acceptance Criteria
- [ ] docs/ui/providers.md exists and covers all six content areas
- [ ] The contract boundary (generated-only guarantee) is stated explicitly
- [ ] The admin module exception is noted
