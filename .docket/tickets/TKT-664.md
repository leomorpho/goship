# TKT-664: eject profile and other app-domain flows from the framework battery surface

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-664` before committing. -->

## Description
GoShip should not treat profile-driven social app flows as framework batteries. This ticket moves app-domain packages like profile out of the productized battery surface and rewires framework batteries to depend on core seams instead of app-specific concepts.

Deliver:
1. Profile is reclassified as starter/example app code rather than framework battery inventory
2. Battery and framework surfaces stop depending on profile-specific concepts by default
3. Any remaining useful generic pieces are extracted to core seams or new batteries
4. Docs/tooling stop presenting app-domain modules as canonical framework batteries

Likely paths:
- modules/profile
- auth/notifications integrations currently keyed to profile
- docs/reference and architecture docs describing first-party modules
- starter/example app surfaces if they need a new home

Out of scope: deleting app-domain behavior that still belongs in a starter app.

Verify commands:
- ship describe --pretty
- ship verify --profile strict
