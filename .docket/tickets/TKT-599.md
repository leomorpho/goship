# TKT-599: [Flags Admin UI] Detect and mark orphaned DB flags

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-599` before committing. -->

## Description
Objective: In the admin flags list, visually identify DB rows that have no matching entry in the code registry — flags that were once defined in code but have since been removed.

Context:
The syncer (TKT-595) deliberately never deletes DB rows when a flag is removed from the registry. This preserves operator-set enabled states and avoids data loss. But it means the DB can accumulate stale rows over time. The admin UI should surface these so an operator can decide to clean them up manually.

An "orphaned" flag is one where the DB row's key has no matching `FlagDefinition` in the registry (i.e. `flags.Lookup(key)` returns `false`).

UI treatment for orphaned rows:
- Render with a distinct visual style (e.g. muted/greyed row, warning badge "Orphaned — not in code").
- Show the DB key and enabled state so the operator can see what it was.
- Do NOT provide an auto-delete button — deletion is always a manual, deliberate action via the existing admin delete flow.
- Orphaned rows appear at the bottom of the list, below all registry-matched rows.

Likely files:
- modules/admin/routes.go — update flags list handler to classify rows as registered vs orphaned
- modules/admin/views/flags.templ (or equivalent) — add orphaned row rendering variant
- modules/flags/registry.go — Lookup() used to classify rows

Likely symbols:
- FlagsListHandler
- FlagRow view model (add IsOrphaned bool field)
- flags.Lookup

Tests to add or update:
- modules/admin/routes_test.go:
  - TestFlagsListHandler_MarksOrphanedRows: DB row with key not in registry renders with "Orphaned" label
  - TestFlagsListHandler_OrphansAtBottom: orphaned rows are ordered after registered rows
  - TestFlagsListHandler_NoOrphans: all DB rows have registry matches → no orphan label rendered

Verify commands:
- docket validate TKT-599

- go test ./modules/admin/... -count=1

Out of scope:
- Auto-deleting orphaned flags.
- Bulk-cleanup tooling (a future CLI concern).
- Any changes to the syncer or evaluator logic.
