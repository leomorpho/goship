# TKT-665: centralize first-party battery installation and process wiring

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-665` before committing. -->

## Description
Web, worker, and test composition currently reconstruct first-party module services in multiple places. That duplication will keep battery migrations drifting. This ticket creates a shared installation layer for first-party batteries so processes compose the same contracts consistently.

Deliver:
1. Shared battery installation/composition functions for web, worker, and test contexts
2. Reduced duplication across cmd/web, cmd/worker, and framework test helpers
3. Battery process-specific hooks remain explicit without reimplementing the same construction logic

Likely paths:
- cmd/web/main.go
- cmd/worker/main.go
- framework/testutil/http.go
- framework/bootstrap/ shared module install wiring

Out of scope: introducing a control-plane architecture; this is local repo composition cleanup.

Verify commands:
- go test ./cmd/...
- go test ./framework/...
- go test ./...

## Acceptance Criteria
- [ ] web, worker, and test processes use shared first-party battery installation logic
- [ ] duplication of module construction across process entrypoints is substantially reduced
- [ ] process-specific behavior remains explicit while battery runtime deps stay consistent
- [ ] go test ./... passes
