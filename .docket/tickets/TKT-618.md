# TKT-618: Promote ship-mcp and MCP docs from optional/future-facing to first-class GoShip LLM workflow guidance.

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-618` before committing. -->

## Description
Objective: Promote ship-mcp and MCP docs from optional/future-facing to first-class GoShip LLM workflow guidance.

Context:
GoShip says it wants advanced LLM support, but the current MCP docs still describe `ship-mcp` as optional and future-facing. Once the repo-local execution model and core tool surfaces are solid, the docs should treat MCP as a supported workflow rather than an afterthought.

Rails/Laravel DX bar:
- Official tooling should be documented like a first-class path when the framework depends on it for its LLM story.
- Humans and LLMs should both be able to discover the supported workflow from entry docs alone.

Likely paths:
- docs/reference/02-mcp.md
- MCP_TOOLS.md
- README.md
- docs/00-index.md
- tools/mcp/ship/README.md

Likely symbols:
- `ship-mcp`
- supported install workflow
- recommended LLM dev loop
- repo-local CLI resolution expectations

Tests to add or update:
- Doc-sync tests if MCP docs or README become part of the enforced canonical set.
- Any README/MCP doc consistency tests introduced by the doc drift gate.

Verify commands:
- docket validate TKT-614
- go test ./tools/cli/ship/internal/commands ./tools/mcp/ship/... -count=1
- sed -n '1,240p' docs/reference/02-mcp.md

Out of scope:
- Implementing new MCP features outside the already-defined scope.
- Landing-page marketing work.
- Generic AI-agent guidance that does not touch the GoShip workflow directly.
