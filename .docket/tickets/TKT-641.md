# TKT-641: fix: push notification delivery blocks request goroutine — enqueue as background jobs

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-641` before committing. -->

## Description
Production bug. PublishNotification() calls both PWA and FCM push services synchronously in the request goroutine. Each call blocks waiting for an HTTP/gRPC response, and iterates over all subscriptions/tokens. The TODO has 6 exclamation marks.

Affected lines:
- push_notifications_pwa.go:91 — webpush.SendNotification() in a loop, blocks per subscription
- push_notifications_fcm.go:123 — fcmClient.Send() in a loop, blocks per token

Fix:
1. Add jobs core.Jobs to NotifierService struct and constructor
2. Define a DeliverPushNotificationJob task type (payload: profileID, notification data, platform)
3. In PublishNotification(), replace direct push calls with jobs.Enqueue()
4. Implement the job handler that calls the appropriate push service
5. Register the handler in the worker process (cmd/worker)
6. Update module.go wiring to pass jobs to NotifierService

Likely paths:
- modules/notifications/notifier.go
- modules/notifications/push_notifications_pwa.go
- modules/notifications/push_notifications_fcm.go
- modules/notifications/module.go
- cmd/worker/main.go (register job handler)

Out of scope: changes to job queue infrastructure, changes to permissions logic.

Verify commands:
- go build ./...
- go test ./modules/notifications/...
- Manual: trigger a notification, confirm request returns immediately, worker log shows push delivery

## Acceptance Criteria
- [ ] PublishNotification() does not call push services synchronously
- [ ] a job is enqueued per push delivery instead
- [ ] job handler calls the appropriate push service (PWA or FCM)
- [ ] handler registered in cmd/worker
- [ ] NotifierService accepts core.Jobs as a constructor dependency
- [ ] request latency is not affected by push delivery time
- [ ] go test ./modules/notifications/... passes
