# TKT-663: replace the notifications monolith with smaller battery packages

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-663` before committing. -->

## Description
Do not continue treating modules/notifications as the future architecture. Replace the monolith with a family of smaller batteries that separate inbox persistence, preference policy, and delivery transports.

Initial target batteries:
- inbox
- notificationpreferences
- webpush
- mobilepush

Potential later batteries:
- reminders
- phoneverification

Deliver:
1. Define the replacement package boundaries and public contracts
2. Port only the useful behavior from the legacy notifications package
3. Keep delivery async and sender adapters pure
4. Remove profile/onboarding/app coupling from the notification architecture

Likely paths:
- new battery directories under modules/
- legacy modules/notifications as migration/reference source
- router/bootstrap/module docs/tooling that enumerate first-party batteries

Out of scope: preserving the current notifications package shape for compatibility if it conflicts with the new boundaries.

Verify commands:
- go test ./modules/...
- ship verify --profile strict

## Acceptance Criteria
- [ ] the future notification surface is split into smaller batteries instead of one monolith
- [ ] ported notification behavior is limited to pieces that still fit the new battery boundaries
- [ ] push/mobile sender adapters are pure delivery layers and do not own permission policy
- [ ] notification architecture no longer depends on app-domain module concepts like profile ownership by default
- [ ] ship verify --profile strict passes
