# TKT-597: [Flags Layer D] Implement GET /api/me/capabilities endpoint

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-597` before committing. -->

## Description
Objective: Add a `GET /api/me/capabilities` endpoint that evaluates all registered flags for the authenticated user and returns a flat `map[string]bool`. The frontend consumes this once on load and never queries flags directly.

Context:
The frontend currently has no clean way to know which features are enabled for the current user. Adding a capabilities endpoint avoids per-feature round-trips and keeps flag evaluation server-side. The response shape is intentionally simple — a flat object of boolean capabilities — so the frontend can do `if (caps.aiEnabled)` without knowing anything about flags, rollouts, or user targeting.

Response shape:
```json
GET /api/v1/me/capabilities
Authorization: <session>

200 OK
{
  "aiEnabled": true,
  "newDashboard": false,
  "betaExport": false
}
```

Key naming: convert the `FlagKey` value (e.g. `"ai.enabled"`) to camelCase for the JSON response (e.g. `"aiEnabled"`) using a simple dot-to-camelCase transform.

Implementation notes:
- Endpoint is authenticated — return 401 if no session.
- Iterate `flags.All()` to get the registered set (not the raw DB rows) — this guarantees only code-defined flags appear in the response.
- For each flag, call `service.Enabled(ctx, key, userID)` to evaluate with user targeting and rollout.
- Response is JSON; keep it cacheable by the client (short-lived, e.g. 60s cache-control).

Route wiring note:
The flags module currently has no route registration of its own — all routes go through the main router. The capabilities handler must be registered in `router.go` under the existing `v1` group (`/api/v1`). The flags module exposes the handler; `router.go` wires it.
