# TKT-178: Make paidsubscriptions integration key-agnostic

## Description
Make paid subscriptions integration key-agnostic by removing module dependency on app plan constants.

Context:
`modules/paidsubscriptions/routes` imports app-level plan keys, violating module isolation and flexibility.

Scope:
- Replace direct free/pro key assumptions with catalog/service-driven checks.
- Remove module import of `app/subscriptions`.

Out of scope:
- Product pricing changes.

Likely paths:
- `modules/paidsubscriptions/routes/routes.go`
- `cmd/web/main.go`
- `cmd/worker/main.go`

Verify commands:
- `go test ./modules/paidsubscriptions/...`
- `go test ./cmd/web ./cmd/worker`

## Acceptance Criteria
- [x] Module routes/services no longer import app/subscriptions constants. : Removed app/subscriptions import from modules/paidsubscriptions/routes; module now uses service methods for free/default-paid semantics.
- [x] Plan checks rely on catalog/service semantics : Pricing and webhook plan selection now route through Service.FreePlanKey/IsPaidPlanKey/DefaultPaidPlanKey catalog semantics.
- [x] not fixed key names. : No fixed free/pro constants are used in module route logic; plan-state checks use catalog metadata.
- [x] Tests verify custom key compatibility. : Added service tests covering custom keys (business/starter/enterprise) for default paid key and paid-plan detection.
