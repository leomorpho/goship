# TKT-640: decouple permissions logic from push notification services

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-640` before committing. -->

## Description
Both PwaPushService and FcmPushService directly call notificationPermissionService.HasPermissionsForPlatform(). The code has TODOs marking this as bad design — push services should be pure senders that know nothing about permissions.

Affected TODOs:
- push_notifications_pwa.go:146 — HasPermissionsLeftAndEndpointIsRegistered()
- push_notifications_fcm.go:155 — HasPermissionsLeftAndTokenIsRegistered()

Fix: move permission checks up to NotifierService.PublishNotification(). Check permissions before dispatching to push services. Push services become pure senders — given subscriptions/tokens, they send.

Likely paths:
- modules/notifications/push_notifications_pwa.go
- modules/notifications/push_notifications_fcm.go
- modules/notifications/notifier.go
- modules/notifications/module.go (wiring update — push services no longer need permission service)

Out of scope: changes to the permissions service itself, DB schema changes.

Verify commands:
- go build ./...
- go test ./modules/notifications/...

## Acceptance Criteria
- [ ] PwaPushService has no dependency on notificationPermissionService
- [ ] FcmPushService has no dependency on notificationPermissionService
- [ ] permission checks exist in NotifierService.PublishNotification() before dispatching to push services
- [ ] module.go wiring updated — push services no longer receive permission service
- [ ] behaviour unchanged: notifications only sent when permissions allow
- [ ] go test ./modules/notifications/... passes
